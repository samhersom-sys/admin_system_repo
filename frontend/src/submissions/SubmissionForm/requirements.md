# SubmissionForm Component — Requirements

## Purpose

`SubmissionForm` is a controlled form component used on the New Submission page.
It collects the minimum data required to create a submission record.
It renders all editable fields, validates them, and calls back to the parent on submit.

It does **not** make API calls.  All data fetching and saving is handled by the page.

## Scope

**In scope:**
- Rendering all required and optional creation fields
- Client-side field validation before calling `onSubmit`
- Insured party search modal integration (open/close, selection populates name + ID)
- Expiry date auto-population when inception date is set and expiry is blank
- Read-only display of reference (system-generated) and status (always 'Created')
- Loading state on the submit button while the parent is saving

**Out of scope:**
- API calls (handled by the page)
- Navigation (handled by the page)
- Reference number generation (handled by the domain)

## Requirements

### R01 — Renders all creation fields
The form must render inputs for every required and optional creation field.

Acceptance criteria:
- Renders a labelled input for **Insured Name** (text, with a search icon button)
- Renders a labelled date input for **Inception Date** (required)
- Renders a labelled date input for **Expiry Date** (required, auto-populated)
- Renders a labelled `<select>` for **Contract Type** with options:
  `Open Market`, `Binding Authority Contract`, `Lineslip`, `Facultative`
- Renders a read-only field displaying **Reference** (value from `reference` prop)
- Renders a read-only field displaying **Status** (always `'Created'`)

### R02 — Inception date auto-populates expiry
When the user sets the inception date and the expiry date is currently empty, the expiry date
must automatically populate to inception + 1 year.

Acceptance criteria:
- User sets inception to `2026-03-10` with expiry blank → expiry becomes `2027-03-10`
- If expiry is already filled in, it must not change when inception changes
- Auto-populated expiry can be manually overridden by the user

### R03 — Validation before submit
The form must prevent submission and display inline errors if required fields are missing.

Acceptance criteria:
- Submit with **Insured Name** empty → error message `"Insured name is required"` visible, `onSubmit` not called
- Submit with **Inception Date** empty → error message `"Inception date is required"` visible, `onSubmit` not called
- Submit with **Expiry Date** empty → error message `"Expiry date is required"` visible, `onSubmit` not called
- Submit with all required fields filled → `onSubmit` is called with the form values, no errors shown

### R04 — Insured party search integration
The search icon on the Insured Name field must open a party search modal. 
Selecting a party from the modal must populate both the name field and the hidden ID field.

Acceptance criteria:
- Clicking the search icon renders `PartySearchModal` (or equivalent) in an open state
- Selecting a party: the Insured Name field populates with `party.name`
- Selecting a party: the hidden `insuredId` field value is set to `party.id`
- The modal can be dismissed without selecting (no change to form state)

### R05 — Loading state
While `isLoading` prop is `true`, the Submit button must be disabled and show a loading indicator.

Acceptance criteria:
- `isLoading={true}` → submit button has `disabled` attribute
- `isLoading={true}` → button label changes from "Create Submission" to "Saving…"
- `isLoading={false}` → submit button is enabled with label "Create Submission"

### R06 — Error display
When `errorMessage` prop is provided, it must be displayed in the form.

Acceptance criteria:
- `errorMessage="Server error"` → text `"Server error"` is visible in the form
- `errorMessage={undefined}` → no error message rendered

## Dependencies

- `PartySearchModal` (or a stub) — for insured party lookup
- Receives `reference` string prop (generated by page before render)
- Receives `onSubmit` callback
- Receives `isLoading` boolean
- Receives `errorMessage` string | undefined
